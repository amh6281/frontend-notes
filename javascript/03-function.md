# Functions

JavaScript의 함수 개념과 특징을 정리한다.

함수는 작업을 수행하거나 값을 계산하는 과정을 하나의 블록으로 감싸 **실행 단위**로 만들어 놓은 것이다.

```ts
function add(x, y) {
  return x + y;
}
```

---

## 함수는 일급 객체다

JavaScript에서 함수는 **일급 객체(first-class object)** 다.

> **일급 객체**: 다른 값에 적용 가능한 연산을 모두 지원하는 객체로, **함수를 숫자나 문자열 같은 일반 값과 똑같이 다룰 수 있다**는 뜻이다.

```ts
const x = 10; // 1. 변수에 담기
doSomething(10); // 2. 함수의 인수로 넘기기
return 10; // 3. 함수에서 반환하기
const obj = { a: 10 }; // 4. 객체/배열 안에 넣기
```

이것이 값에 적용 가능한 연산으로, 함수가 일급 객체라는 것은 **`10`이 들어갈 수 있는 자리에 함수도 들어갈 수 있다**는 말이다.

```ts
const add = function (x, y) {
  return x + y;
}; // 1. 변수에 담기

[1, 2, 3].map((item) => item * 2); // 2. 인수로 넘기기

function multiplier(n) {
  return (x) => x * n; // 3. 함수를 반환하기
}

const calculator = { add }; // 4. 객체에 넣기
```

#### 왜 "일급(first-class)"이라는 이름이 붙었는가?

모든 언어가 함수를 값처럼 다루는 것은 아니다. 함수를 일반 값과 동등하게(일등 시민처럼) 취급한다는 의미에서 **일급**이라고 부른다.

#### 함수가 일급 객체라서 무엇이 가능해지는가?

- 인수로 넘길 수 있다 → **콜백 패턴**, `map` 같은 **고차 함수**
- 반환할 수 있다 → **커링**, **클로저 활용**
- 변수·객체에 담을 수 있다 → 함수를 조합하는 **함수형 프로그래밍 스타일**

React의 컴포넌트, 커스텀 훅, HOC(고차 컴포넌트)도 모두 이 성질 위에 세워져 있다. `React.memo(Component)`처럼 컴포넌트(함수)를 다른 함수에 넘기는 것이 가능한 이유가 바로 함수가 값이기 때문이다.

## 함수를 정의하는 방식

### 함수 선언식 (Function Declaration)

```ts
function add(x, y) {
  return x + y;
}
```

### 함수 표현식 (Function Expression)

함수를 값으로 만들어 변수에 할당하는 방식이다. 함수가 일급 객체이기 때문에 가능하다.

```ts
const add = function (x, y) {
  return x + y;
};
```

---

## 함수 선언식 vs 함수 표현식 — 호이스팅 차이

**호이스팅(hoisting)**: 변수 및 함수 선언이 해당 스코프의 최상단으로 끌어올려진 것처럼 동작하는 것

- 함수 선언식: **함수 전체**가 호이스팅된다 → 선언 이전에 호출 가능
- 함수 표현식: **변수 선언만** 호이스팅되고, 함수 할당은 실행 시점에 일어난다 → 선언 이전에 호출 불가

```ts
// 함수 선언식
add(2, 3); // 5 — 호출 가능

function add(x, y) {
  return x + y;
}
```

```ts
// 함수 표현식
subtract(5, 3); // ReferenceError: Cannot access 'subtract' before initialization

const subtract = function (x, y) {
  return x - y;
};
```

#### 함수 표현식도 "호이스팅이 아예 안 되는" 것은 아니다

정확히는 **변수 선언 자체는 호이스팅된다.** 다만 어떤 키워드로 선언했는지에 따라 에러의 종류가 달라진다.

- `var`로 선언: 변수가 `undefined`로 초기화된 채 호이스팅 → 호출 시 `TypeError: subtract is not a function`
- `const`/`let`으로 선언: 호이스팅은 되지만 **TDZ(Temporal Dead Zone)** 에 놓여 초기화 전 접근 불가 → `ReferenceError`

> 즉 "함수 표현식은 호이스팅 X"보다는 "**함수 할당이 런타임에 일어나므로 선언 이전에는 사용할 수 없다**"가 더 정확한 표현이다.

#### 무엇을 쓰는 것이 좋은가?

정답은 없지만, 보통 함수 표현식(const + 화살표 함수)을 선호한다.

- 선언 이전에는 호출할 수 없어 코드가 위 → 아래 순서로 읽힌다
- const라 재할당을 막을 수 있다

반대로 함수 선언식은 호이스팅되므로, 유틸 함수를 파일 하단에 두고 상단에서 먼저 쓰는 구성이 가능하다.

## 화살표 함수 (Arrow Function)

ES6에서 도입된 함수 정의 방식으로, 일반 함수와 문법만 다른 것이 아니라 **동작 자체가 다르다.**

```ts
const add = (x, y) => x + y;
```

일반 함수와의 차이는 다음과 같다.

- `constructor`가 없다 → `new`로 호출할 수 없다
- `arguments` 객체가 없다
- **자신의 `this`를 가지지 않는다** → 언제나 상위 스코프의 `this`를 그대로 사용한다
- `prototype` 프로퍼티가 없다

#### 왜 화살표 함수는 자신의 this를 가지지 않는가?

일반 함수의 `this`는 **호출될 때마다 동적으로 결정**된다. 이 때문에 콜백 안에서 `this`가 의도와 다르게 바뀌는 문제가 자주 생겼다.
화살표 함수는 이를 해결하려고 `this`를 아예 만들지 않는다. 대신 **선언된 위치의 상위 스코프 `this`를 그대로 사용**한다. 호출 방식과 무관하게 고정되는 이 방식을 **lexical this**라고 한다.

```ts
class Counter {
  count = 0;

  increaseWrong() {
    setTimeout(function () {
      this.count += 1; // this가 Counter 인스턴스가 아님
    }, 1000);
  }

  increaseRight() {
    setTimeout(() => {
      this.count += 1; // 상위 스코프(increaseRight)의 this를 그대로 사용
    }, 1000);
  }
}
```

#### 화살표 함수를 쓰면 안 되는 곳은?

`this`가 동적으로 결정되어야 하는 자리에는 쓰면 안 된다.

- **객체의 메서드**: 화살표 함수로 정의하면 `this`가 그 객체가 아니라 상위 스코프(보통 전역)를 가리킨다
- **생성자 함수**: `new`로 호출할 수 없다
- **`addEventListener` 콜백에서 `this`로 이벤트 대상 요소를 쓰려는 경우**

```ts
const counter = {
  count: 0,
  increase: () => {
    this.count += 1; // this는 counter가 아님
  },
};
```

## 일반 함수에서 this는 어떻게 결정되는가?

**this**: 자신이 속한 객체나 자신이 생성할 인스턴스를 가리키는 값

일반 함수의 `this`는 함수가 **어떻게 호출되었는지**에 따라 동적으로 결정된다. 크게 4가지 규칙이 있다.

| 호출 방식                  | this가 가리키는 것                         |
| -------------------------- | ------------------------------------------ |
| 일반 함수로 호출           | 전역 객체 (strict mode에서는 `undefined`)  |
| 메서드로 호출 (`obj.fn()`) | 메서드를 호출한 객체 (`obj`)               |
| 생성자로 호출 (`new fn()`) | 새로 생성되는 인스턴스                     |
| `call` / `apply` / `bind`  | 첫 번째 인수로 전달한 객체 (명시적 바인딩) |

```ts
const obj = {
  name: "Lee",
  getName() {
    return this.name;
  },
};

obj.getName(); // "Lee", 메서드 호출, this는 obj

const fn = obj.getName;
fn(); // undefined, 일반 호출, this는 전역 객체(strict mode면 에러)

fn.call(obj); // "Lee", 명시적 바인딩
```

> 같은 함수라도 **호출 방식이 바뀌면 `this`도 바뀐다**는 것이 핵심이다. 화살표 함수는 이 규칙에서 완전히 제외되며, 선언 시점의 상위 스코프 `this`로 고정된다.

## arguments와 rest parameter

여러 개의 인수를 한꺼번에 받는 방법을 다룬다.

### arguments (옛날 방식)

`arguments`는 일반 함수 안에서 자동으로 만들어지는, 전달된 모든 인수를 담고 있는 객체다. 매개변수를 적지 않아도 넘어온 인수가 전부 여기에 들어간다.

```ts
function sum() {
  // arguments = 넘긴 인수 전부 (예: sum(1, 2, 3) → [1, 2, 3])
  return Array.from(arguments).reduce((acc, cur) => acc + cur, 0);
}
```

두 가지 불편함이 있다.

- 유사 배열이라 진짜 배열이 아니다. `map`, `reduce` 같은 배열 메서드를 바로 쓸 수 없어 `Array.from()`으로 변환해야 한다.
- 화살표 함수에는 없다. 화살표 함수에서 `arguments`를 쓰면 상위 스코프의 것을 참조해버린다.

### rest parameter (권장 방식)

rest parameter(`...args`)는 매개변수 앞에 `...`을 붙여, 넘어온 인수들을 하나의 배열로 묶어 받는 문법이다. 현재는 `arguments` 대신 이 방식이 권장된다. 처음부터 진짜 배열로 들어오므로 배열 메서드를 바로 쓸 수 있고, 화살표 함수에서도 동작한다.

```ts
const sum = (...args) => args.reduce((acc, cur) => acc + cur, 0);
```

## 다양한 함수 살펴보기

### 즉시 실행 함수 (IIFE, Immediately Invoked Function Expression)

함수를 정의하는 순간 즉시 실행되는 함수. 한 번 실행된 후 다시 호출할 수 없다.

```ts
(function (a, b) {
  return a + b;
})(10, 24); // 34
```

#### 즉시 실행 함수는 언제 쓰는가?

내부 변수를 외부에 노출하지 않는 **독립적인 스코프**를 만들 수 있다. ES6 이전에는 전역 오염을 막는 사실상 유일한 수단이었다.

> 지금은 블록 스코프(`let`/`const`)와 모듈 시스템이 스코프 격리를 대신하기 때문에 사용 빈도가 줄었다. 다만 "초기화 로직을 한 번만 실행하고 흔적을 남기지 않는다"는 의도를 드러내는 용도로는 여전히 쓰인다.

### 고차 함수 (Higher-Order Function)

함수를 인수로 받거나, 결과로 새로운 함수를 반환하는 함수. 함수가 일급 객체이기 때문에 가능하다.

```ts
// 함수를 인수로 받는 대표적인 고차 함수: Array.prototype.map
const doubleArray = [1, 2, 3].map((item) => item * 2);

doubleArray; // [2, 4, 6]
```

```ts
// 함수를 반환하는 고차 함수
const add = (a) => (b) => a + b;

add(1)(3); // 4
```

#### React에서의 고차 함수

이 개념을 컴포넌트에 적용한 것이 **고차 컴포넌트(HOC, Higher-Order Component)** 다. 컴포넌트를 인수로 받아 새로운 컴포넌트를 반환하며, `React.memo`가 대표적인 예다. 커스텀 훅과 함께 로직 재사용 수단으로 쓰인다.

## 함수를 만들 때 주의해야 할 사항

### 1. 부수 효과(side effect)를 최대한 억제하기

**부수 효과**: 함수가 자신의 스코프 밖에 있는 값을 변경하거나, 외부 세계에 영향을 주는 것 (외부 변수 수정, DOM 조작, API 호출, 콘솔 출력 등)

부수 효과가 없고 **같은 입력에 대해 항상 같은 출력을 반환하는 함수**를 **순수 함수(pure function)** 라고 한다. 순수 함수는 예측 가능하고 테스트하기 쉽다.

```ts
// ❌ 부수 효과: 외부 상태를 직접 수정
let total = 0;
function addToTotal(x) {
  total += x;
}

// ✅ 순수 함수: 입력만으로 출력이 결정됨
function add(a, b) {
  return a + b;
}
```

> 모든 부수 효과를 없앨 수는 없다(API 호출, 렌더링 자체가 부수 효과다). 핵심은 부수 효과를 **없애는 것**이 아니라 **예측 가능한 곳으로 격리**하는 것이다. React가 `useEffect`라는 이름으로 부수 효과를 한 곳에 모으게 하는 것도 같은 맥락이다. 또한 React의 렌더링 단계에서 컴포넌트(함수)는 순수해야 한다는 규칙도 여기서 나온다.

### 2. 가능한 한 함수를 작게 만들기

함수가 하나의 일만 하도록 만든다. 함수가 커질수록 코드 냄새(code smell)가 생길 확률이 높아지고, 내부에서 무슨 일이 일어나는지 추적하기 어려워진다.

### 3. 누구나 이해할 수 있는 이름 붙이기

함수 이름만 보고도 어떤 동작을 하는지 알 수 있어야 한다. 간결하고 명확하게 짓는다. (예: `useEffect` 콜백에 이름을 붙여 어떤 효과인지 드러내는 것도 한 방법이다.)

## 요약

- 함수는 **일급 객체**다 → 변수 할당, 인수 전달, 반환이 모두 가능하고 이것이 고차 함수·콜백·HOC의 기반이 된다.
- 함수 선언식은 **함수 전체가 호이스팅**되지만, 함수 표현식은 할당이 런타임에 일어나 선언 이전에 호출할 수 없다 (`const`/`let`이면 TDZ로 인한 `ReferenceError`).
- 일반 함수의 `this`는 **호출 방식**에 따라 동적으로 결정되고, 화살표 함수는 자신의 `this` 없이 **상위 스코프의 `this`를 정적으로** 사용한다.
- 화살표 함수는 `constructor`, `arguments`, `prototype`이 없다 → 메서드·생성자로는 부적합하다.
- 함수는 **부수 효과를 격리**하고, **작게**, **명확한 이름**으로 만든다.
