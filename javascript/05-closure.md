# Closure

클로저와 React가 이를 활용하는 방식을 정리한다.

---

## 클로저란

**함수가 만들어질 당시 주변에 있던 변수들을 기억하고, 나중에도 계속 쓸 수 있는 것.**

즉, 함수가 자기 바깥의 변수에 접근할 수 있는 상태를 계속 유지하는 것

```ts
const createCounter = () => {
  let count = 0;

  const increment = () => {
    count++; // 바깥의 count를 사용
    console.log(`현재 카운트: ${count}`);
  };

  return increment;
};

const counter = createCounter();
counter(); // 현재 카운트: 1
counter(); // 현재 카운트: 2
```

보통 함수는 실행이 끝나면 내부 변수가 사라진다. 그런데 `createCounter()`는 이미 끝났는데도 `count`는 살아남아 계속 올라간다.

`increment`가 자기가 태어난 곳(`createCounter` 안)의 `count`를 붙들고 있기 때문이다. 그래서 `counter()`를 부를 때마다 그 `count`에 접근해 값을 바꾼다.

#### 렉시컬(어휘적) 환경?

함수가 **선언된 위치 주변에 어떤 변수들이 있었는지** 기록해 둔 메모

- **렉시컬(lexical)** = "코드에 쓰여 있는 위치 그대로"라는 뜻
- 즉 함수를 **어디서 호출했는지**가 아니라, **어디에 적어놨는지**로 결정된다

```ts
const createCounter = () => {
  let count = 0; // increment가 선언된 자리 주변에 count가 있음

  const increment = () => {
    // → increment는 "내 바깥에 count가 있다"를 기억한 채 태어남
    count++;
  };

  return increment;
};
```

`increment`를 어디로 넘겨서 언제 호출하든, 기억하는 `count`는 처음 선언될 때 주변에 있던 그 `count`다.

> **함수는 자기가 쓰여진 자리를 기억하고, 그 자리의 변수를 계속 쓸 수 있다.**

#### 렉시컬 스코프 vs 동적 스코프

상위 스코프를 **무엇을 기준으로 정하느냐**에 따라 두 방식으로 나뉜다.

- **렉시컬 스코프** — 함수를 **어디에 선언했는지**로 상위 스코프가 정해진다.
- **동적 스코프** — 함수를 **어디서 호출했는지**로 상위 스코프가 정해진다.

**JavaScript는 렉시컬 스코프를 쓴다.** 클로저가 성립하는 근거도 바로 이것이다.

#### 왜 함수가 끝났는데 변수가 안 사라지는가?

JavaScript 엔진의 **가비지 컬렉터(GC)** 는 "아직 이 변수를 쓰는 누군가가 있는지" 확인하고, 아무도 안 쓰는 메모리만 정리한다.

`createCounter`가 끝나도 반환된 `increment`가 `count`를 참조하고 있어서, GC가 `count`(정확히는 그 렉시컬 환경)를 지우지 못한다.

클로저가 "발생 이유"는 **참조가 남아 있어 렉시컬 환경이 회수되지 못하기 때문.**

## 스코프

변수가 유효한 범위.

### 전역 스코프 (Global Scope)

가장 바깥, 전역 레벨에 선언한 것. 어디서든 접근할 수 있다.

```ts
var globalValue = "전역";

const fn = () => {
  console.log(globalValue); // "전역" (어디서든 접근 가능)
};
```

### 함수 스코프 (Function Scope)

함수 안에서 선언한 변수는 그 함수 안에서만 유효하다. `var`가 여기에 해당한다.

```ts
var x = 10;

const number = () => {
  var x = 100;
  console.log(x); // 100 (자기 스코프의 x)

  const number2 = () => {
    var x = 1000;
    console.log(x); // 1000
  };
  number2();
};

number();
console.log(x); // 10 (전역의 x는 그대로)
```

안쪽에서 바깥 변수는 볼 수 있지만, 바깥에서 안쪽 변수는 못 본다.

### 블록 스코프 (Block Scope)

`{}` 블록 단위로 유효 범위가 잘린다. ES6의 `let`, `const`가 여기에 해당한다.

```ts
if (true) {
  var varValue = "var";
  let letValue = "let";
}

console.log(varValue); // "var"  (var는 블록을 무시)
console.log(letValue); // ReferenceError (let은 블록 밖에서 못 씀)
```

`var`는 블록을 스코프로 치지 않는다. `if`, `for` 같은 블록 안에 선언해도 밖으로 새어 나온다. 뒤에 나올 반복문 문제의 원인이 바로 이것이다.

### 스코프 체인

변수를 찾을 때 자기 스코프에서 시작해 바깥으로 한 단계씩 올라간다. 전역까지 갔는데도 없으면 에러다.

```
increment 안에서 count를 찾음
  └─ 없음 → 바깥(createCounter)에서 찾음 → 있음! 사용
              └─ 없었다면 → 전역에서 찾음
                             └─ 없으면 ReferenceError
```

클로저는 이 체인이 함수 종료 후에도 유지되는 것이라고 보면 된다.

> **프로토타입 체이닝**과 헷갈리기 쉽다. 프로토타입 체이닝은 **객체의 메서드/프로퍼티**를 찾아 올라가는 것이고, 스코프 체인은 **변수**를 찾아 올라가는 것이다.

### 실행 컨텍스트와 렉시컬 환경

렉시컬 환경은 **실행 컨텍스트** 안에 들어 있다.

- **실행 컨텍스트(Execution Context)** — 코드가 실행될 때 만들어지는 정보 묶음. 변수·함수 선언, 스코프 체인, `this` 바인딩 등을 담는다. 전역 코드가 실행되면 전역 컨텍스트가, 함수가 호출되면 함수 컨텍스트가 생겨 콜스택에 쌓인다.
- **렉시컬 환경(Lexical Environment)** — 그 컨텍스트 안에서 **현재 스코프의 변수/함수 정보 + 상위 스코프 참조**를 보관하는 부분.

함수가 끝나 컨텍스트가 콜스택에서 빠진 뒤에도 내부 함수가 그 환경을 참조하고 있으면 클로저가 된다.

```
실행 컨텍스트 (호출 시 생성)
  └─ 렉시컬 환경 (변수 + 상위 스코프 참조)
        └─ 내부 함수가 이걸 계속 붙들고 있음 → 클로저
```

## React에서의 클로저

```tsx
const Counter = () => {
  const [count, setCount] = useState(0);

  const handleClick = () => {
    setCount(count + 1); // 바깥 스코프의 count에 접근
  };

  return <button onClick={handleClick}>{count}</button>;
};
```

`handleClick`은 자기 바깥에 있는 `count`를 기억한다. 나중에 버튼이 클릭돼서 실행될 때도 그 `count`를 쓴다.

#### useState 함수 호출은 어디에 있는가?

첫 줄의 `useState(0)`이 호출이다. 이 줄에서 호출이 시작되고 끝나며, `[count, setCount]` 배열을 반환한다.

호출은 첫 줄에서 이미 끝났는데, `setCount`는 어떻게 계속 최신 state를 알고 값을 바꿀 수 있을까?

`setCount`가 `useState` 안에서 만들어진 함수이기 때문이다. `useState`의 호출이 끝나도, `setCount`는 자기가 태어난 `useState` 내부의 환경(state가 보관된 곳)을 기억하고 있다.

```
useState 호출 → 끝남
      └─ 하지만 반환된 setCount가 내부 state를 붙들고 있음
             → 나중에 호출해도 그 state에 접근 가능
```

`createCounter`가 끝나도 `increment`가 `count`를 계속 썼던 것과 똑같은 구조다.

#### 왜 setCount(count + 1)보다 함수형 업데이트가 안전한가?

`handleClick`이 기억하는 `count`는 **그 함수가 만들어진 시점의 값**으로 고정돼 있다. 그래서 연속으로 실행하면 같은 값을 기준으로 계산한다.

```ts
setCount(count + 1);
setCount(count + 1); // count가 0이면 둘 다 0 + 1 → 결과는 2가 아니라 1
```

이렇게 옛날 값을 붙들고 있는 상황을 **stale closure(오래된 클로저)** 라고 부른다. 이때는 최신 값을 인수로 받는 함수형 업데이트를 쓴다.

```ts
setCount((prev) => prev + 1);
setCount((prev) => prev + 1); // 0 → 1 → 2
```

## 주의: 반복문과 var

```ts
for (var i = 0; i < 5; i++) {
  setTimeout(() => {
    console.log(i);
  }, i * 1000);
}
```

`0, 1, 2, 3, 4`를 기대하지만 **5가 5번** 출력된다.

#### 왜 5만 5번 나오는가?

`var`는 블록 스코프가 없어서 `i`가 반복문 바깥에 딱 하나만 존재한다. 5개의 콜백이 각자 값을 복사해 가는 게 아니라, **똑같은 `i` 하나를 다 같이 보고 있는 것**이다.

`setTimeout` 콜백이 실행될 시점에는 반복문이 이미 다 끝나서 `i`는 `5`다. 그래서 다섯 개 모두 5를 찍는다.

#### 해결 1. let으로 바꾸기

`let`은 블록 스코프라, 반복마다 `i`가 새로 만들어진다. 각 콜백이 각자의 `i`를 기억한다.

```ts
for (let i = 0; i < 5; i++) {
  setTimeout(() => {
    console.log(i); // 0, 1, 2, 3, 4
  }, i * 1000);
}
```

#### 해결 2. 클로저 사용하기

즉시 실행 함수로 감싸서 반복마다 별도의 스코프를 만든다. 그 시점의 `i`를 `j`로 복사해 가둬두는 방식이다.

```ts
for (var i = 0; i < 5; i++) {
  ((j) => {
    setTimeout(() => {
      console.log(j); // 0, 1, 2, 3, 4
    }, j * 1000);
  })(i);
}
```

> `let`이 훨씬 간결하다. 클로저 방식은 `let`이 없던 시절의 해법이라고 보면 된다.

## 클로저는 어디에 쓰나

- **데이터 은닉(캡슐화)**: 바깥에서 직접 못 건드리는 변수를 만든다. `createCounter`의 `count`는 반환된 함수를 통해서만 바뀐다. 전역 변수를 줄이고 상태를 안전하게 지킬 수 있다.
- **상태 유지**: 함수 호출이 끝나도 값을 기억한다. React의 `useState`가 state를 붙들고 있는 것과 같은 원리다.
- **함수형 패턴**: 특정 값을 미리 가둬둔 함수를 만들 수 있다(부분 적용, 커링 등).

```ts
const createCounter = () => {
  let count = 0; // 바깥에서 직접 접근 불가, 은닉됨

  return {
    increment: () => ++count,
    get: () => count,
  };
};

const c = createCounter();
c.increment(); // 1
c.get(); // 1
// count에 직접 손댈 방법이 없다 → 캡슐화
```

## 클로저를 남용하면 안 되는 이유

클로저에는 비용이 따른다. 생성될 때마다 자신이 선언된 환경을 기억해야 하기 때문이다.

기억한다는 건 그 변수들을 메모리에 계속 붙들고 있다는 뜻이다. 함수가 끝나도 정리되지 않는다. 꼭 필요한 것만 남기지 않으면 메모리를 불필요하게 잡아먹는다.

> 클로저를 쓰지 말라는 얘기가 아니다. 무거운 값을 습관적으로 가둬두지 말고, 필요한 만큼만 기억시키라는 뜻이다.

## 요약

- **클로저**는 함수가 자기가 선언된 자리의 변수를 기억하고 계속 쓰는 것. 내부 함수가 참조를 붙들고 있어 **가비지 컬렉터**가 그 렉시컬 환경을 회수하지 못해 발생한다.
- **렉시컬 스코프**는 "어디에 선언했는지" 기준으로 상위 스코프를 정한다. 호출 위치 기준인 동적 스코프의 반대이고, JS는 렉시컬 스코프다.
- **스코프**는 전역 / 함수(`var`) / 블록(`let`, `const`)으로 나뉘고, 변수는 스코프 체인을 따라 바깥으로 찾아 올라간다.
- **활용**은 데이터 은닉(캡슐화), 상태 유지, 함수형 패턴. React의 `useState`도 클로저다.
- 옛날 값을 붙드는 **stale closure**를 만나면 `setCount(prev => prev + 1)`를 쓴다.
- `var`는 블록 스코프가 없어 반복문에서 하나의 변수를 공유한다. `let`으로 해결한다.
- 클로저는 환경을 메모리에 붙들고 있으므로 필요한 만큼만 쓴다.
